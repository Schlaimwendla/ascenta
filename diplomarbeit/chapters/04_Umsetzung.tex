\subsubsection*{Projektkonfiguration (Gradle und Manifest)}
Die grundlegende Architektur, das Abhängigkeitsmanagement und die Rechteverwaltung der Android-Applikation werden über zwei zentrale Steuerungsdateien definiert.

\textbf{Abhängigkeitsmanagement (build.gradle.kts)} \\
Das \textit{Gradle-Build-System} automatisiert den Kompilierungsprozess und verwaltet externe Bibliotheken \cite{android_gradle}. In der \texttt{build.gradle.kts} auf App-Ebene wird die minimale Kompatibilität (\texttt{minSdk = 31}) auf Android 12 festgelegt. Dies ist eine bewusste Designentscheidung, da erst ab dieser Version die entkoppelten, modernen Bluetooth-Berechtigungen zur Verfügung stehen. Zudem werden hier die essenziellen Kernbibliotheken importiert, darunter die \textit{Vosk Offline Engine} (\texttt{vosk-android:0.3.47}) für die lokale Spracherkennung sowie \texttt{viewBinding = true} zur typsicheren Anbindung der grafischen XML-Layouts an den Kotlin-Code.

\textbf{Rechteverwaltung und Deklaration (AndroidManifest.xml)} \\
Die Datei \texttt{AndroidManifest.xml} fungiert als Steuerzentrale gegenüber dem Betriebssystem \cite{android_manifest}. Sie registriert die Kernkomponenten der Applikation, primär die \texttt{MainActivity} und den Netzwerk-Dienst (\texttt{AscentaService}). 
Ein kritischer Aspekt ist die Deklaration der Hardware-Berechtigungen (Permissions). Da das System aktiv nach BLE-Geräten sucht und TCP-Verbindungen aufbaut, fordert das Manifest spezifische Netzwerk- und Bluetooth-Rechte an. Ausschlaggebend ist hierbei die Nutzung der neuen Android 12 Bluetooth-Rechte (\texttt{BLUETOOTH\_SCAN} und \texttt{BLUETOOTH\_CONNECT}), welche die feingranulare Kopplung der Brille erlauben. Um den TCP-Stream bei deaktiviertem Bildschirm aufrechtzuerhalten, wird der \texttt{AscentaService} explizit mit dem Attribut \texttt{foregroundServiceType="connectedDevice"} beim System deklariert.

\subsubsection*{Android Lifecycle-Management und Systementkopplung}
Da das Smartphone während der Nutzung der Assistenzbrille typischerweise in der Tasche des Nutzers verbleibt und der Bildschirm deaktiviert ist, neigt das Android-Betriebssystem dazu, Hintergrundprozesse aus Energiespargründen zu terminieren (Doze-Mode). Die Architektur lagert daher die gesamte Netzwerk- und Bluetooth-Logik in den deklarierten \texttt{AscentaService} aus. 

Durch die Eskalation dieses Dienstes zu einem \textit{Foreground Service} priorisiert das Betriebssystem den Prozess dauerhaft auf die höchste Stufe \cite{android_foreground}. Das Android-System verlangt für diese Priorisierung zwingend die Einbindung einer permanenten Benachrichtigung (Notification), welche dem Anwender die aktive Nutzung im Hintergrund signalisiert:

\begin{lstlisting}[language=Java, caption={Start des Foreground Services zur Verhinderung der Terminierung}]
val notification = NotificationCompat.Builder(this, CHANNEL_ID)
    .setContentTitle("Ascenta Assistenzsystem")
    .setContentText("Netzwerkverbindung zur Brille aktiv...")
    .setSmallIcon(R.drawable.ic_eye)
    .setPriority(NotificationCompat.PRIORITY_HIGH)
    .build()

// Zwingt Android, den Service bei Display-Off nicht zu killen
startForeground(1, notification)
\end{lstlisting}

\subsubsection*{Asynchrone BLE-GATT State-Machine}
Das System überträgt die WLAN-Credentials (Provisioning) initial über das Bluetooth Low Energy (BLE) Generic Attribute Profile (GATT) vom Smartphone auf die Brille. Die größte programmiertechnische Herausforderung hierbei ist die stark ereignisgesteuerte, asynchrone API des Android-Bluetooth-Stacks \cite{android_gatt}. Die Applikation implementiert hierfür eine strikte State-Machine innerhalb eines \texttt{BluetoothGattCallback}.

Um Puffer-Überläufe auf dem Bluetooth-Chip des Nicla Vision zu verhindern, darf das System SSID und Passwort nicht simultan senden. Die Kotlin-Logik schreibt zunächst die SSID in die entsprechende Characteristic und wartet zwingend auf den \texttt{onCharacteristicWrite}-Callback mit dem Status \texttt{GATT\_SUCCESS}, bevor das Passwort kaskadierend übertragen wird:

\begin{lstlisting}[language=Java, caption={Kaskadierendes Provisioning der WLAN-Daten via BLE-GATT Callback}]
override fun onCharacteristicWrite(
    g: BluetoothGatt, 
    char: BluetoothGattCharacteristic, 
    status: Int
) {
    if (status == BluetoothGatt.GATT_SUCCESS) {
        if (char.uuid == SSID_CHAR_UUID) {
            // SSID erfolgreich bestaetigt, nun Passwort uebermitteln
            val passChar = g.getService(PROV_SERVICE_UUID).getCharacteristic(PASS_CHAR_UUID)
            passChar.value = pendingPassword!!.toByteArray()
            g.writeCharacteristic(passChar)
            
        } else if (char.uuid == PASS_CHAR_UUID) {
            // Provisioning komplett abgeschlossen
            g.disconnect()
            startTcpConnection() // Einleitung der Hauptkommunikation
        }
    }
}
\end{lstlisting}

\subsubsection*{Evolution der Netzwerkarchitektur (BLE $\rightarrow$ UDP $\rightarrow$ TCP)}
Die finale Implementierung der Netzwerktopologie resultierte aus einer dreistufigen Evaluierungsphase während der Entwicklung. Da das System hohe Anforderungen an Latenz und Datensicherheit stellt, dokumentiert der folgende Prozess die Abwägung und praktische Erprobung verschiedener Protokollstacks.

\vspace{1em}
\noindent\textbf{Phase 1: Das initiale BLE-GATT-Profil} \\
Das ursprüngliche Design sah das Nicla Vision als eigenständiges BLE-Peripheral vor. Das System spannte einen GATT-Server auf, um Daten über spezifische Characteristics zu übertragen. Die Übertragung von Audiodaten (ca. $128.000$ Bytes) stieß jedoch auf massive Barrieren. BLE-Pakete sind im Standard auf eine Payload (MTU) von 20 Bytes limitiert. Um den Hardware-Puffer des Mikrocontrollers zu schützen und einen \texttt{OSError 11} zu vermeiden, musste der Versand künstlich auf 15\,ms Intervalle gedrosselt werden. Dies führte zu inakzeptablen Latenzen von mehreren Sekunden für eine einzige Sprachnachricht.

\begin{lstlisting}[language=Python, caption={Berechnung des Audiospeichers und initiales BLE-Setup}]
# 16000 Hz * 1 Channel * 2 Bytes * 4.0s = 128.000 Bytes Pufferbedarf
TOTAL_BYTES = int(SAMPLE_RATE * CHANNELS * BYTES_PER_SAMPLE * RECORD_SECONDS)
UART_SERVICE_UUID = bluetooth.UUID("6E400001-B5A3-F393-E0A9-E50E24DCCA9E")
\end{lstlisting}

\vspace{1em}
\noindent\textbf{Phase 2: Der Zwischenschritt über das User Datagram Protocol (UDP)} \\
Um die BLE-Limitierung zu umgehen, erfolgte in der Entwicklung der Wechsel auf ein WiFi-basiertes Netzwerk unter Nutzung von UDP. Dieses Protokoll ermöglichte theoretisch eine verzögerungsfreie Übertragung, da es keine Bestätigungszyklen (Acknowledge) erfordert. In der praktischen Erprobung offenbarte UDP jedoch eine kritische Instabilität: \textit{Packet Loss}. In einem mobilen Szenario, in dem sich der Nutzer bewegt, führten verlorene Datenpakete zu korrupten JPEG-Dateien und unvollständigen Audio-Chunks. Da UDP keine Fehlerkorrektur bietet, konnte die Applikation die defekten Binärströme nicht mehr zusammensetzen.

\vspace{1em}
\noindent\textbf{Phase 3: Finale hybride Topologie (BLE \& TCP)} \\
Aufgrund der Unzuverlässigkeit von UDP wechselten wir final zum Transmission Control Protocol (TCP). TCP garantiert durch interne Prüfsummen und automatische Paketwiederholung eine verlustfreie Übertragung. Das System nutzt in der finalen Umsetzung nun eine hybride Strategie: BLE übernimmt ausschließlich die Geräteerkennung und den Austausch der WLAN-Daten (Provisioning). Nach erfolgreichem BLE-Provisioning verbindet sich der Mikrocontroller mit dem WLAN und die App etabliert daraufhin den Hauptdatenstrom über eine verbindungsorientierte TCP-Verbindung (Port 5005). Die Firmware konfiguriert den Socket auf dem Nicla Vision mittels \texttt{setblocking(False)}. Dies verhindert, dass Netzwerkverzögerungen den Mikrocontroller in einen Deadlock versetzen.

\subsubsection*{Stream-Multiplexing und Custom Protocol Parser}
Die Nutzung einer einzigen TCP-Verbindung (Port 5005) für völlig unterschiedliche Datentypen kennzeichnet eine architektonische Besonderheit des Systems. Der Datenstrom (\textit{Stream}) interleavt kontinuierliche Textnachrichten (JSON-Telemetrie) mit großen binären Payloads (JPEG-Bilder, WAV-Audiodaten). 

Ein eigenes Applikationsprotokoll (Application Layer) verhindert eine Vermischung der Datentypen. Der Code stellt binären Übertragungen einen definierten Präfix inklusive Payload-Größe voran (z.\,B. \texttt{AUD\_START:12500}). Der Android \texttt{AscentaService} implementiert einen maßgeschneiderten Parser auf Basis von Kotlin-Coroutines, der den Stream zunächst zeichenweise bis zum Line-Feed (\texttt{\textbackslash n}) ausliest und bei Erkennung eines Binär-Präfixes temporär auf einen blockierenden Byte-Reader umschaltet:

\begin{lstlisting}[language=Java, caption={Kotlin-Coroutine für dynamisches TCP-Stream-Multiplexing und Thread-Switching}]
tcpJob = CoroutineScope(Dispatchers.IO).launch {
    val input = tcpSocket!!.getInputStream()
    while (isActive && isConnected) {
        val line = readLineFromStream(input) // Liest ASCII bis zum Line-Feed \n
        
        if (line.startsWith("IMG_START:")) {
            val size = line.substringAfter(":").toInt()
            // Temporaerer Wechsel auf blockierenden Binär-Reader
            val imgBytes = readBytesFromStream(input, size)
            val bitmap = BitmapFactory.decodeByteArray(imgBytes, 0, imgBytes.size)
            // Synchronisation mit dem Main-Thread fuer UI-Updates
            withContext(Dispatchers.Main) { updateUiWithImage(bitmap) }
            
        } else if (line.startsWith("AUD_START:")) {
            val size = line.substringAfter(":").toInt()
            val audioBytes = readBytesFromStream(input, size)
            processAudioWithVosk(audioBytes)
            
        } else if (line.startsWith("{")) {
            parseTelemetryJson(line) // Uebergabe an Standard JSON-Parser
        }
    }
}
\end{lstlisting}

\subsubsection*{Ausfallsicherheit durch Heartbeat-Mechanismus}
Die Applikation implementiert einen asynchronen Watchdog, um Verbindungsabbrüche zuverlässig zu erkennen. Über den stehenden TCP-Socket sendet die App alle zehn Sekunden einen Ping an die Brille. Bleibt der Response für mehr als 25 Sekunden aus, schließt die Software die Verbindung sicher und reaktiviert den BLE-Discovery-Modus zur automatischen Neuverbindung (\textit{Self-Healing}):

\begin{lstlisting}[language=Java, caption={Heartbeat-Coroutine zur Überwachung der Verbindungsstabilität}]
heartbeatJob = CoroutineScope(Dispatchers.IO).launch {
    while (isActive) {
        if (isConnected) {
            sendCommand("ping")
            delay(10000)
            if (System.currentTimeMillis() - lastPingResponse > 25000) {
                withContext(Dispatchers.Main) {
                    isConnected = false
                    stopHeartbeat()
                    startBleScan() // Zurueck in den Discovery-Status
                }
            }
        } else { delay(5000) }
    }
}
\end{lstlisting}

\subsubsection*{Concurrency und Thread-Sicherheit in Kotlin}
Die Android-Applikation führt netzwerkbasierte I/O-Operationen und rechenintensive Machine-Learning-Prozesse (Vosk) simultan aus. Dafür implementiert der Code ein striktes Thread-Dispatching mittels \textit{Coroutines}. Sämtliche Sockets und blockierende Lesevorgänge operieren ausschließlich im Hintergrund-Thread (\texttt{Dispatchers.IO}). Sobald die Coroutine ein Datenpaket vollständig empfängt, führt sie einen synchronisierten Kontextwechsel zurück auf den Main-Thread durch (\texttt{withContext\allowbreak(Dispatchers.\allowbreak Main)}), um die Benutzeroberfläche zu aktualisieren oder die TTS-Engine konfliktfrei anzusteuern.

\subsubsection*{Ressourcenmanagement und Modell-Integration (Nicla Vision)}
Die Firmware des Nicla Vision (\texttt{firmware.py}) operiert unter extremen Speicherrestriktionen. Von den theoretischen 2 MB RAM des STM32H7-Prozessors steht MicroPython nach dem Bootvorgang lediglich ein nutzbarer Heap von ca. 1 MB zur Verfügung. Dieser begrenzte Speicher muss gleichzeitig den 115 KB großen Framebuffer der Kamera, die Netzwerk-Sockets, die Audio-Puffer sowie die hochkomplexen Tensoren des Machine-Learning-Modells fassen.

Da die dynamische Speicherallokation in MicroPython bei solch starker Auslastung schnell zu Speicherfragmentierung (Heap Fragmentation) führt, was einen Systemabsturz zur Folge hat, wenn kein zusammenhängender Speicherblock für Bild-Tensoren mehr gefunden wird, greift die Architektur auf proaktive Maßnahmen zurück. Die Firmware reserviert initial einen Notfall-Puffer über \texttt{micropython.alloc\_emergency\_exception\_buf(100)}, um Abstürze während hardwarenaher Interrupts abzufangen. Zudem erzwingt der Code eine deterministische Garbage-Collection-Strategie. Der Befehl \texttt{gc.collect()} wird strategisch nach jedem Frame-Snapshot manuell aufgerufen, um nicht mehr referenzierte Objekte sofort zu vernichten und der Fragmentierung entgegenzuwirken.

\subsubsection*{Die Kernschleife der Umgebungserfassung}
Die Hauptmethode \texttt{run\_detection} steuert die kontinuierliche Überwachung der Umgebung. Der Kamerasensor wird auf \texttt{sensor.RGB565} und \texttt{sensor.QVGA} konfiguriert. Ein anschließendes \texttt{windowing((240, 240))} schneidet das Bild exakt auf die benötigte Input-Tensor-Größe zu. Erkennt der Algorithmus ein valides Objekt, konstruiert die Firmware dynamisch einen JSON-String mit Label, Konfidenzwert und Distanz und streamt diesen asynchron über den TCP-Socket:

\begin{lstlisting}[language=Python, caption={Initialisierung des Sensors und Extraktion der Vorhersagewerte}]
def run_detection():
    gc.collect() # Proaktive Defragmentierung
    try:
        sensor.set_windowing((240, 240)) # Exakter Zuschnitt auf Model-Input
        img = sensor.snapshot()
        
        # Inferenz des FOMO-Modells
        result = g_model.predict([img])[0].flatten().tolist()
        best_index = max(range(len(result)), key=lambda i: result[i])
        best_conf = result[best_index]
        
        label_name = LABELS[best_index] if best_index < len(LABELS) else "Unknown"
        distance_mm = tof.read() if tof else 0
        
        del img # Sofortige Freigabe des 115 KB Framebuffers
        gc.collect()
        
        if best_conf >= 0.6:
            json_str = json.dumps({
                "label": label_name,
                "conf": f"{best_conf:.2f}",
                "dist": distance_mm
            }) + "\n"
            send_tcp_packet(json_str.encode())
            
    except Exception as e:
        print(f"Vision Err: {e}")
\end{lstlisting}

\subsubsection*{Auswertung des 4D-Tensors und Sensorfusion}
Die Auswertung der Vorhersagedaten des FOMO-Modells stellt eine besondere algorithmische Herausforderung dar. Das quantisierte Modell generiert einen vierdimensionalen Tensor (Heatmap), der das Bild in ein Grid unterteilt. Die Firmware iteriert über das X/Y-Grid. Findet der Algorithmus ein lokales Maximum über dem Schwellenwert, berechnet die Firmware den absoluten Bildschwerpunkt (Zentroid) und fusioniert die 2D-Bildkoordinaten mit der Z-Achse (Distanz in Millimetern) des ToF-Sensors. Diese fusionierten Raumkoordinaten werden unmittelbar im Anschluss in das JSON-Format serialisiert und asynchron über den TCP-Stream an das Smartphone gesendet. Dort dienen sie der Applikation als Berechnungsgrundlage für die präzise, direktionale Sprachausgabe an den Nutzer.

\subsubsection*{Low-Level Hardware-Kommunikation (I2C \& Registerauslesung)}
Eine verlässliche Batterieanzeige ist für blinde Nutzer essenziell. Die Firmware liest den integrierten Fuel-Gauge-IC (MAX17262) direkt über den I2C-Bus aus. Der Mikrocontroller adressiert den Chip unter der Hexadezimal-Adresse \texttt{0x36} und fordert zwei Bytes ab dem Register \texttt{0x06} (RepSOC) an. Da der Chip die Daten hardwareseitig im \textit{Little-Endian}-Format überträgt, konvertiert die Software die rohen Bytes in einen Integer und dividiert den Wert anschließend durch $256,0$, um den Ladezustand in Prozent zu erhalten:

\begin{lstlisting}[language=Python, caption={Direkte I2C-Registerauslesung des State-of-Charge (SOC) vom MAX17262}]
def get_soc(self):
    if not self.fuel_gauge_i2c: return None
    try:
        # Auslesen von 2 Bytes aus dem RepSOC Register (0x06) des ICs
        data = self.fuel_gauge_i2c.readfrom_mem(0x36, 0x06, 2)
        
        # Konvertierung des Little-Endian Bytearrays zu einem Integer
        soc_raw = int.from_bytes(data, 'little')
        
        # Berechnung des realen Prozentwerts gemaess Datenblatt-Faktor
        return soc_raw / 256.0 
    except:
        return None
\end{lstlisting}

\subsubsection*{Kinetische Interaktionssteuerung (Tap-Detection)}
Zur Freihandbedienung (Aktivierung der Spracherkennung) liest die Firmware den internen 6-Achsen-IMU-Sensor (LSM6DSOX) via I2C aus. Um zuverlässig einen beabsichtigten Doppeltipp (Double-Tap) von der normalen Gehbewegung zu unterscheiden, wertet der Algorithmus den \textit{Ruck} ($\Delta g$) aus.

Überschreitet der kombinierte Beschleunigungsvektor einen empirisch ermittelten Schwellenwert von $1,75$ G, speichert das System den Zeitstempel. Folgt ein zweiter Ausschlag in einem definierten Zeitfenster zwischen 100 und 600 Millisekunden, wird die Aktion als valider Double-Tap registriert und das Kommando an das Smartphone gesendet:

\begin{lstlisting}[language=Python, caption={Filterung von Störbewegungen und Berechnung der Differenzbeschleunigung}]
x, y, z = self.lsm.accel()
# Berechnung des Rucks (Aenderungsrate der Beschleunigung)
dx = abs(x - self.last_accel[0])
dy = abs(y - self.last_accel[1])
dz = abs(z - self.last_accel[2])
self.last_accel = (x, y, z)

delta_g = dx + dy + dz # Kombinierter Beschleunigungsvektor

if delta_g > 1.75:
    now = time.ticks_ms()
    diff = time.ticks_diff(now, self.last_tap_time)
    
    # Pruefung, ob der zweite Tap im validen 600ms Fenster erfolgte
    if 100 < diff < 600:
        self.send_tcp_packet(b'{"action": "tap"}\n')
        self.req_audio = True 
        self.last_tap_time = 0
    elif diff > 600:
        self.last_tap_time = now # Ersten Tap als Startpunkt setzen
\end{lstlisting}

\subsubsection*{Speichereffizientes Audio-Streaming}
Die kontinuierliche Aufzeichnung von Audiodaten stellt eine hohe Auslastung für den begrenzten Arbeitsspeicher dar. Eine 3,5-sekündige Audioaufnahme bei 16\,kHz beansprucht exakt $112.000$ Bytes. Die Firmware fragmentiert den Puffer daher in Arrays zu je 4096 Bytes. Ein hardware-getriebener Callback füllt diese Chunks iterativ. 

Durch den gezielten Einsatz von \texttt{memoryview} wird eine \textit{Zero-Copy}-Zuweisung erzwungen: MicroPython verschiebt die Audio-Bytes direkt in die vorallozierten Speicheradressen, anstatt bei jeder Zuweisung neuen Arbeitsspeicher anzufordern. Nach Abschluss generiert die Firmware dynamisch einen 44 Byte großen RIFF/WAVE-Header mittels \texttt{struct.pack} und streamt diesen direkt mit den Daten über den TCP-Socket:

\begin{lstlisting}[language=Python, caption={Zero-Copy Speicherzuweisung und dynamische Generierung des WAV-Headers}]
def create_header(data_size):
    # Bytegenaues Packing gemaess der offiziellen WAV/RIFF-Spezifikation
    return struct.pack('<4sI4s4sIHHIIHH4sI', 
                       b'RIFF', 36+data_size, b'WAVE', b'fmt ', 16, 1,
                       config.CHANNELS, config.SAMPLE_RATE, 
                       config.SAMPLE_RATE * config.CHANNELS * 2, 
                       config.CHANNELS * 2, 16, b'data', data_size)

def _audio_callback(self, buf):
    mv = memoryview(buf)
    l = len(mv)
    off = 0
    while l > 0:
        chunk = g_chunks[self.rec_idx]
        space = AUDIO_CHUNK_SIZE - self.rec_offset
        amt = min(l, space)
        
        # Zero-Copy Zuweisung in den pre-allozierten Buffer
        chunk[self.rec_offset : self.rec_offset+amt] = mv[off : off+amt]
        self.rec_offset += amt
        off += amt
        l -= amt
\end{lstlisting}

\subsubsection*{Dynamisches Asset-Management des Akustikmodells}
Ein spezifisches technisches Hindernis unter Android besteht darin, dass kompilierte C++-Bibliotheken (Vosk JNI) keine Dateien direkt aus dem komprimierten APK-Asset-Ordner lesen können. Sie erfordern absolute Dateipfade auf einem unkomprimierten Dateisystem \cite{vosk_api}. Die Applikation löst dieses Problem durch einen asynchronen Entpackungsprozess beim Erststart. Eine IO-Coroutine iteriert rekursiv durch das Modell-Verzeichnis der Assets und kopiert die binären Akustik-Modelle Byte für Byte in den internen \texttt{filesDir}-Speicherbereich des Smartphones.

\subsubsection*{Audio-Verarbeitung und kognitive Entlastung}
Nach der Transkription des Audiosignals oder der Objekterkennung erfolgt die Sprachausgabe über die offizielle \texttt{TextToSpeech}-API von Android \cite{android_tts}. Um den blinden Nutzer nicht mit Dauerwarnungen zu überlasten, implementiert die Applikation eine dynamische Cooldown-Logik. Befindet sich das Objekt in einer unkritischen Distanz (über 100\,cm), wird ein Cooldown von sechs Sekunden für dieses spezifische Label aktiviert. Unterschreitet die vom ToF-Sensor gemessene Distanz jedoch einen kritischen Schwellenwert (z.\,B. unter 40\,cm), greift eine Override-Logik, die den Cooldown ignoriert und eine sofortige Kollisionswarnung erzwingt.

\subsubsection*{I/O-Auslagerung beim Bild-Streaming}
Die Brille bietet zur Verifikation des Sichtfeldes die Funktion, hochauflösende Snapshots an die Smartphone-App zu senden. Ein unkomprimiertes Bitmap belegt jedoch den verbleibenden RAM nahezu vollständig. 

Zur Lösung dieses Speicherengpasses nutzt die Firmware ein klassisches \textbf{Swap-Verfahren} auf dem Dateisystem. Das aufgenommene Bild wird direkt unter Anwendung einer JPEG-Kompression auf den Flash-Speicher der Brille geschrieben (\texttt{temp.jpg}). Unmittelbar danach zerstört die Firmware das rechenintensive Bildobjekt im RAM manuell durch den \texttt{del img}-Befehl und erzwingt eine Garbage-Collection. Erst mit dem nun wieder freien Arbeitsspeicher wird die Datei \texttt{temp.jpg} stückweise ausgelesen und an das Handy gestreamt:

\begin{lstlisting}[language=Python, caption={Verwendung des Flash-Speichers als Swap zur Vermeidung von OOM-Exceptions}]
def process_image(self):
    path = "temp.jpg"
    try:
        img = sensor.snapshot()
        img.save(path, quality=75) # Speichern im Flash
        
        del img       # Manuelle RAM-Freigabe des 115 KB Bitmaps
        gc.collect()  # Zwingende Defragmentierung
        
        size = os.stat(path)[6]
        self.send_tcp_packet(f"IMG_START:{size}\n".encode())
        
        # Speicherschonendes, stueckweises Auslesen der Swap-Datei
        with open(path, 'rb') as f:
            while True:
                chunk = f.read(1024)
                if not chunk: break
                self.send_tcp_packet(chunk)
                
        uos.remove(path) # Temporäre Datei sicher löschen
    except Exception as e:
        print(f"Img Err: {e}")
    finally:
        gc.collect()
\end{lstlisting}

\subsubsection*{Schaltungskonzept und Energieversorgung}
Die zentrale Entscheidung beim Schaltungsdesign war die Aufteilung der Energiequelle. Anstatt einen großen, schweren Akku zu verbauen, wurden zwei identische LiPo-Zellen \cite{eemb_akku_620mah} mit einer Kapazität von jeweils 620\,mAh parallelgeschaltet. 
Der wesentliche Vorteil dieser Parallelschaltung besteht darin, dass die Spannung bei nominal 3,7\,V stabil bleibt, was exakt den Betriebsspezifikationen der Nicla Vision entspricht. Gleichzeitig addiert sich die Gesamtkapazität auf 1240\,mAh, sodass die Brille länger genutzt werden kann, ohne dass ein einzelner Akku zu groß für den Brillenbügel wird. Durch diese Aufteilung auf zwei Zellen wird zudem der Entladestrom pro Akku halbiert. Dies verringert die thermische Belastung der einzelnen Zellen und erhöht langfristig die Lebensdauer.

Ein wichtiger Faktor bei der Planung war die Entladecharakteristik der LiPo-Zellen. Wie aus dem typischen Spannungsverlauf (siehe Abbildung \ref{fig:entladungskurve}) ersichtlich wird, fällt die Spannung nach einem initialen Abfall über einen langen Zeitraum nur minimal ab (stabile Entladephase), bevor sie gegen Ende rapide einbricht.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.7\textwidth]{media/images/entladungskurve_lipo.png}
    \caption{Typische Entladungskurve eines Lithium-Polymer-Akkus}
    \label{fig:entladungskurve}
\end{figure}

Aufgrund dieser flachen Kurve ist es technisch kaum möglich, den exakten Ladezustand (z.\,B. 70\,\% vs. 30\,\%) allein über eine Spannungsmessung zu ermitteln. Um zu verhindern, dass die Brille am kritischen Endpunkt der Kurve ohne Vorwarnung abschaltet, macht sich das System den in die Nicla Vision integrierten Fuel-Gauge-Chip (MAX17262) zunutze. Dieser misst den exakten Stromverbrauch und berechnet die verbleibende Kapazität präzise.

\subsubsection*{Integration des Sicherheitsmanagements}
Um das theoretische Risiko von Akkus in unmittelbarer Kopfnähe zu minimieren, wurde ein TP4056-Lademodul \cite{tp4056_datasheet} als zentrale Kontrollinstanz zwischen die Energiespeicher und die restliche Elektronik geschaltet (siehe Abbildung \ref{fig:tp4056}):
\begin{itemize}
    \item \textbf{Schutzfunktion:} Das Modul überwacht den Stromfluss permanent. Bei einem Kurzschluss oder einer Überlastung trennt es die physische Verbindung zu den Akkus augenblicklich.
    \item \textbf{Laderegelung:} Das Aufladen erfolgt sicher über einen integrierten USB-C-Anschluss. Das Modul steuert exakt, wann der Akku vollständig geladen ist, um eine Überladung zu verhindern. Zudem vereinfacht es den Ladevorgang maßgeblich, da beide Zellen durch die Parallelschaltung gleichzeitig aufgeladen werden können.
\end{itemize}

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.5\textwidth]{media/images/tp4056.png}
    \caption{Das verwendete TP4056-Lademodul}
    \label{fig:tp4056}
\end{figure}

\subsubsection*{Lademodul (TP4056)}
Zur sicheren und unkomplizierten Aufladung der Akkus fiel die Wahl auf das TP4056 Lademodul. Da die Brille unabhängig von speziellen Ladegeräten funktionieren soll, stellte ein Modul mit Standard-USB-C-Anschluss die optimale Lösung dar. Das TP4056 ist äußerst kompakt und lässt sich dadurch platzsparend in der Glashalterung platzieren, ohne das Design der Brille zu beeinträchtigen. Es fungiert als zentrale Schnittstelle zwischen einer externen Stromquelle und den internen LiPo-Akkus.

Wichtigste Funktionen für das Projekt:
\begin{itemize}
    \item \textbf{Sicheres Laden:} Das Modul regelt den Ladevorgang nach dem CC/CV-Verfahren. Die Akkus werden zuerst mit konstantem Strom und anschließend mit konstanter Spannung geladen, was die Lebensdauer der Zellen verlängert und Überhitzung verhindert.
    \item \textbf{Tiefentladeschutz:} Eine wesentliche Sicherheitsfunktion ist der Schutz vor Tiefentladung. Das Modul trennt die Verbindung zwischen Akku und Nicla Vision automatisch, bevor die Zellspannung einen kritischen Wert unterschreitet. Dies verhindert dauerhafte Schäden an den empfindlichen LiPo-Akkus.
    \item \textbf{Überlastschutz:} Sollte das System durch einen Fehler zu viel Strom ziehen oder ein Kurzschluss entstehen, schaltet das Modul den Ausgang sofort ab. Da die Elektronik direkt am Kopf getragen wird, ist diese hardwareseitige Sicherung für die Nutzersicherheit unverzichtbar.
\end{itemize}
Integration in das System: Das Lademodul ist so verbaut, dass der USB-C-Port von außen leicht zugänglich bleibt. Von dieser Position aus wird der Ladestrom an beide parallelgeschalteten Akkus verteilt. Gleichzeitig erfolgt die Versorgung der Nicla Vision über den Out-Pin mit der nötigen Betriebsspannung. Durch diese Bauweise wird eine saubere Trennung zwischen dem Ladevorgang und dem eigentlichen Betrieb der Brille realisiert.

Der Schaltplan zeigt die Vernetzung der zentralen Komponenten für das Energiemanagement der Brille.

    \caption{Schaltplan der zentralen Energieversorgung und Verdrahtung}
    \label{fig:schaltplan}

Zentrale Verdrahtung:

\subsubsection*{Eingesetzte Plattformen}
Für die Datenverarbeitung sowie die Modellentwicklung kamen folgende Plattformen zum Einsatz:
\begin{itemize}
    \item \textbf{Edge Impulse:} Edge Impulse ist eine führende Entwicklungsumgebung für Machine Learning auf Embedded-Geräten (TinyML). Die Plattform ermöglicht den gesamten Workflow von der Datenerfassung über die Merkmalsextraktion bis hin zum Training optimierter Modelle.
    \item \textbf{Roboflow:} Roboflow ist ein spezialisiertes Tool für das Management von Computer-Vision-Datensätzen. Es unterstützt Entwickler dabei, Bilder effizient zu annotieren, zu organisieren und durch Augmentierung künstlich zu erweitern. Zudem bietet es Schnittstellen, um die vorbereiteten Daten direkt in verschiedene Trainings-Frameworks zu exportieren.
    \item \textbf{Kaggle:} Kaggle ist die weltweit größte Online-Community für Data Science und Machine Learning. Die Plattform bietet Zugriff auf eine riesige Auswahl an öffentlichen Datensätzen. Darüber hinaus stellt sie kostenlose cloudbasierte Rechenleistung (GPUs) zur Verfügung, um komplexe Modelle direkt im Browser zu trainieren.
\end{itemize}

\subsubsection*{Architektur des Neuronalen Netzes}
Die Plattform Edge Impulse diente der Konstruktion und dem Training des künstlichen neuronalen Netzes (Convolutional Neural Network, kurz CNN \cite{wikipedia_cnn}). Edge Impulse automatisiert den Aufbau der komplexen Netzstrukturen, greift im Hintergrund jedoch auf die bewährte Schichtarchitektur für die Bildverarbeitung zurück.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.9\textwidth]{media/images/cnn_neural_network.png}
    \caption{Prinzipieller Aufbau eines Convolutional Neural Networks (CNN)}
    \label{fig:cnn_architektur}
\end{figure}

Aufbau und Schichtmodell in der Verarbeitungskette: Ein in Edge Impulse definierter \glqq Impulse\grqq\ (eine Daten-Pipeline) übernimmt die schrittweise Analyse der Kamerabilder:
\begin{itemize}
    \item \textbf{Eingabeschicht (Input Layer):} Die Eingabeschicht leitet die Kamerabilder in einer vorab festgelegten, reduzierten Auflösung in das Netz ein.
    \item \textbf{Faltungsschichten (Convolutional Layers):} Das Netz wendet in diesen Schichten verschiedene Filter auf das Bild an. Dieser Prozess extrahiert wichtige visuelle Merkmale wie Kanten, Formen und Texturen.
    \item \textbf{Pooling-Schichten (Subsampling):} Diese Schichten reduzieren die räumlichen Dimensionen der Daten innerhalb des Netzes. Dies senkt die Rechenlast für den Mikrocontroller drastisch und behält ausschließlich die relevantesten Informationen bei.
    \item \textbf{Ausgabeschicht (Output Layer):} Die Ausgabeschicht berechnet abschließend die Wahrscheinlichkeiten für die definierten Objektklassen.
\end{itemize}

Einsatz der FOMO-Architektur: Die Konfiguration in Edge Impulse nutzt explizit die FOMO-Architektur (Faster Objects, More Objects). Dieses Framework wurde speziell für leistungslimitierte Hardware wie die Nicla Vision konzipiert.
\begin{itemize}
    \item \textbf{Klassifizierung statt Regression:} Das Verfahren verzichtet bewusst auf die rechenintensive Berechnung von Bounding Boxes (Rechtecken um die Objekte).
    \item \textbf{Grid-basierte Erkennung:} Der FOMO-Algorithmus teilt das Kamerabild in ein Raster auf. Anschließend bewertet das neuronale Netz für jede Zelle einzeln, welches Objekt sich dort mit welcher Wahrscheinlichkeit befindet.
    \item \textbf{Effizienz und Deployment:} Dieser Ansatz erzielt eine enorm hohe Verarbeitungsgeschwindigkeit. Nach dem Training stellt ein direkter Export aus Edge Impulse das fertige Modell als quantisierte TensorFlow Lite Datei (.tflite) bereit, was eine nahtlose Ausführung auf der Hardware der Brille ermöglicht.
\end{itemize}

\subsubsection*{Modelltraining und Parameterkonfiguration}
Nach der Vorverarbeitung der Bilddaten startet in Edge Impulse der eigentliche Trainingsprozess des neuronalen Netzes. Eine gezielte Anpassung der Hyperparameter stellt dabei ein optimales Gleichgewicht zwischen Erkennungsgenauigkeit und Ressourceneffizienz her.

Konfiguration der Trainingsparameter:
\begin{itemize}
    \item \textbf{Trainingszyklen und Lernrate:} Die Festlegung der Epochen (Training cycles) auf 60 gibt dem Algorithmus ausreichend Durchläufe, um die visuellen Muster zu erlernen. Gleichzeitig vermeidet dieser Wert ein sogenanntes \glqq Overfitting\grqq\ (Auswendiglernen der Trainingsdaten). Eine definierte Lernrate (Learning rate) von 0,001 passt die internen Gewichtungen des Netzes in kleinen, präzisen Schritten an.
    \item \textbf{Validierungsset:} Eine gezielte Aufteilung reserviert 20\,\% des gesamten Datensatzes ausschließlich als Validierungsset. Dieses Vorgehen überprüft die reale Performance des Modells kontinuierlich an Bildern, die das Netz im laufenden Training noch nicht gesehen hat.
    \item \textbf{Batch-Größe:} Die definierte Batch-Größe (Batch size) von 32 führt dem Netz jeweils exakt 32 Bilder gleichzeitig zu, bevor der Algorithmus die internen Gewichtungen aktualisiert.
\end{itemize}

    \centering
    \includegraphics[width=0.8\textwidth]{media/images/trainingsgraph.png}
    \caption{Verlauf des Verlustes (Loss) über die Trainings-Epochen}

Spezifikation der Netzwerkarchitektur:
\begin{itemize}
    \item \textbf{Eingabeschicht (Input layer):} Die Eingabeschicht speist exakt 172.800 Features in das neuronale Netz ein. Diese Zahl resultiert direkt aus der gewählten Auflösung und den Farbkanälen der übergebenen Bilddaten.
    \item \textbf{Basis-Modell (Backbone):} Die Architektur nutzt das FOMO-Framework auf Basis eines MobileNetV2 0.35 Modells. Diese spezifische Struktur arbeitet extrem leichtgewichtig und optimiert die Rechenabläufe speziell für Mikrocontroller.
    \item \textbf{Ausgabeschicht (Output layer):} Die Konfiguration richtet die letzte Schicht gezielt darauf aus, exakt eine Klasse (1 class) auszugeben. Das neuronale Netz konzentriert sich durch diese Limitierung ausschließlich auf die Erkennung des definierten Zielobjekts.
\end{itemize}

Optimierung für den Mikrocontroller: Die gezielte Aktivierung der Option \glqq Profile int8 model\grqq\ in den erweiterten Einstellungen bereitet das Modell bereits während der Konfiguration auf die nachfolgende Quantisierung vor. Diese Maßnahme garantiert eine speichereffiziente Übertragung des resultierenden Modells im 8-Bit-Format auf den Flash-Speicher der Brille. Mit diesen Spezifikationen wurde nach dem Training ein F1-Score von 80\,\% erreicht.

\subsubsection*{Softwareseitige Validierung}
Die nachfolgende Analyse schlüsselt die Code-Passagen und ihre technischen Funktionen detailliert auf.

1. Sensor-Konfiguration und Bildvorbereitung: Vor der Dateneinspeisung in das neuronale Netz passt die Initialisierung die Kamera exakt an die Vorgaben des Modells an. Die Konfiguration setzt die Grundauflösung zunächst auf QVGA (320x240 Pixel). Da das FOMO-Modell ein quadratisches Eingangsformat verlangt, schneidet der Befehl \texttt{sensor.set\_windowing} das Bild aktiv auf 240x240 Pixel zu. Abschließend gibt der Parameter \texttt{skip\_frames} der Kamera 2000 Millisekunden Zeit, um den Weißabgleich und die Belichtung an die Umgebung anzupassen.

2. Dynamisches Speichermanagement: Der limitierte Arbeitsspeicher (RAM) der Nicla Vision erzwingt einen effizienten Umgang mit den Ressourcen. Der Code lädt die Modelldatei .tflite direkt in das System. Der Parameter \texttt{load\_to\_fb} (Load to Framebuffer) entscheidet dabei dynamisch über den Speicherort. Eine logische Prüfung weist das System an, das Modell nur dann in den regulären RAM zu laden, wenn danach noch eine Sicherheitsreserve von 64 Kilobyte frei bleibt. Dies verhindert Systemabstürze während der rechenintensiven Bildverarbeitung.

3. Post-Processing und die Heatmap-Übersetzung: Das neuronale Netz liefert keine fertigen Koordinaten, sondern eine vierdimensionale Matrix (Heatmap) mit Wahrscheinlichkeiten. Die Post-Processing-Routine übersetzt diese Matrix in greifbare Objektdaten. Eine Schleife iteriert durch die Ausgabe-Schichten des Netzes. Die Multiplikation der rohen Wahrscheinlichkeitswerte des quantisierten Modells mit 255.0 erzwingt durch die gezielte Fließkomma-Operation ein gültiges Datenformat für die OpenMV-Bibliothek. Dieser Schritt wandelt die Datenmatrix in ein virtuelles Graustufenbild um und nutzt die \texttt{find\_blobs}-Funktion, um die dichtesten Pixel-Zentren (die erkannten Objekte) zu isolieren.

4. Die Hauptschleife: Die Endlosschleife führt die eigentliche Erkennung durch und erzeugt ein visuelles Feedback. Der Sensor erfasst kontinuierlich neue Kamerabilder und übergibt sie über \texttt{net.predict} an das neuronale Netz. Das Programm ignoriert die Klasse "0", da diese standardmäßig den Hintergrund repräsentiert. Für jedes valide Objekt berechnet der Algorithmus den geometrischen Mittelpunkt und zeichnet einen farbigen Kreis in das Live-Bild, um die Funktionalität optisch direkt am Bild zu verifizieren.

\section{Herausforderungen in der Entwicklung}
Die Realisierung eines kompakten, tragbaren Assistenzsystems, welches rechenintensive Machine-Learning-Prozesse mit kontinuierlicher Netzwerkkommunikation vereint, bringt diverse interdisziplinäre Herausforderungen mit sich. 

\subsection{Ressourcenlimits und Speicher-Fragmentierung (Software) - Liam}
Auf softwareseitiger Ebene stellte der extrem begrenzte Arbeitsspeicher (RAM) des Nicla Vision die größte Hürde dar. Die dynamische Speicherallokation in MicroPython führte bei der kontinuierlichen Bildverarbeitung wiederholt zur \textit{Heap-Fragmentierung}. Benötigte das FOMO-Modell anschließend einen zusammenhängenden Speicherblock für den Tensor, schlug die Allokation trotz ausreichendem Gesamtspeicher fehl. Die stringente Einführung der \textit{Zero-Copy}-Verfahren (\texttt{memoryview}), der Auslagerung auf den Flash-Speicher sowie die erzwungene Garbage-Collection lösten diese Instabilitäten endgültig.

\subsection{Softwaretechnische Hürden und Optimierung (Objekterkennung) - Fabian}
Die Implementierung der Objekterkennung auf der Nicla Vision brachte aufgrund genereller Hardware-Limitierungen komplexe Herausforderungen mit sich. Dieser Abschnitt beschreibt die strategischen Entscheidungen und die technischen Lösungen.

\begin{itemize}
    \item \textbf{Speichermangel und Modell-Architektur:} Die stark begrenzte Kapazität des Flash-Speichers verhinderte den Einsatz großer Standard-Architekturen zur Bildklassifizierung. Da solche Modelle den vorhandenen Speicherplatz bei weitem überstiegen, machte dieser Engpass ein eigens trainiertes, schlankes Modell zwingend erforderlich, welches sich exakt an die Ressourcen der Nicla Vision anpasst.
    
    \item \textbf{Auflösung versus Verarbeitungsgeschwindigkeit:} Ein direkter Zusammenhang zwischen der Bildauflösung und der Rechenleistung bestimmte die Performance der Brille. Eine hohe Eingangsauflösung der Kamera resultierte in einer niedrigen Bildrate (FPS), was eine flüssige Objekterkennung in Echtzeit verhinderte. Die Lösung bestand in der Reduzierung der Auflösung sowie der Implementierung des FOMO-Frameworks. FOMO verzichtet auf rechenintensive Bounding Boxes und erkennt Objekte stattdessen ressourcenschonend als Schwerpunkte im Bild.
    
    \item \textbf{Modell-Optimierung durch Quantisierung:} Eine mathematische Optimierung des Netzes nutzt den begrenzten Arbeitsspeicher (RAM) und Flash-Speicher effizient. Durch die sogenannte Quantisierung werden die internen Gewichte des trainierten Modells von 32-Bit-Fließkommazahlen (float32) in 8-Bit-Ganzzahlen (int8) umgewandelt. Dieses Verfahren reduzierte den Speicherverbrauch signifikant und ermöglichte erst die Ausführung des Modells auf dem Mikrocontroller, ohne die Erkennungsrate entscheidend negativ zu beeinflussen.
\end{itemize}

\subsection{Energiemanagement im begrenzten Bauraum - Fabian}
Die größte Hürde bei der Entwicklung von Hilfsmitteln für Menschen mit Sehbehinderung ist der Kompromiss zwischen technischer Leistungsfähigkeit und Alltagstauglichkeit. Während stationäre Systeme oft sehr präzise arbeiten, scheitern mobile Lösungen häufig an ihrer Unhandlichkeit. Im Bereich der Elektronik ergaben sich daraus drei spezifische Herausforderungen:

\begin{itemize}
    \item \textbf{Gewicht VS Tragekomfort:} Eine Brille muss über einen längeren Zeitraum getragen werden können, ohne Druckstellen zu verursachen. Dies limitiert das zulässige Gewicht der Akkus massiv. Gleichzeitig ist die Bildverarbeitung der Nicla Vision durch die integrierte KI rechenintensiv, was den Stromverbrauch im Vergleich zu einfachen Sensoren deutlich erhöht.
    \item \textbf{Elektronische Integration:} Eine signifikante Herausforderung stellte der begrenzte Platz auf den schmalen Brillenbügeln dar. Da die gesamte Elektronik direkt am Gestell untergebracht werden muss, durften die Komponenten die Ergonomie der Brille nicht negativ beeinflussen. Jedes Bauteil musste extrem flach und unauffällig gewählt werden.
    \item \textbf{Sicherheitsaspekte:} Da die Stromversorgung unmittelbar am Kopf getragen wird, ist die Sicherheit ein kritischer Faktor. Bei der Verwendung von Lithium-Polymer-Akkus (LiPo) muss durchgehend sichergestellt sein, dass es unter keinen Umständen zu einer Überhitzung kommt. 
\end{itemize}

Einordnung des Risikos: Trotz dieser theoretischen Gefahr ist das Risiko in der Praxis aus zwei Gründen minimal, weshalb die Entscheidung dennoch auf LiPo-Zellen fiel:
\begin{itemize}
    \item \textbf{Geringe Energiedichte:} Die verwendeten Akkus sind sehr klein. Die darin gespeicherte Gesamtenergie ist im Vergleich zu herkömmlichen Smartphone-Akkus so gering, dass selbst bei einem Defekt kaum genug thermische Energie für eine gefährliche Kettenreaktion vorhanden wäre.
    \item \textbf{Systembedingte Überwachung:} Das Restrisiko wird durch den Einsatz des speziellen Lademoduls adressiert, welches den Stromkreis bei Fehlern sofort physisch trennt.
\end{itemize}

\section{Tests und Validierung - Liam}
Eine Reihe von qualitativen Praxistests belegt die Funktionstauglichkeit des \glqq Ascenta\grqq-Prototyps. Die Evaluierung fokussiert sich auf die Kernanforderungen eines Assistenzsystems: die wahrgenommene Systemlatenz, die Zuverlässigkeit der Sensoreingaben sowie die energetische Stabilität.

\subsection{Netzwerklatenz und Streaming-Performance}
Da es sich bei der Objekterkennung um ein sicherheitskritisches Feature handelt, ist die Verzögerung (Latenz) zwischen physischer Hinderniserkennung und auditiver Warnung ein zentraler Bewertungsfaktor. Praxistests zeigten, dass die gewählte \glqq Split-Brain\grqq-Architektur den Echtzeitanforderungen gerecht wird. Der reine Telemetrie-Datentransfer (JSON) über den etablierten TCP-Stream erfolgt nahezu verzögerungsfrei, sodass Kollisionswarnungen unmittelbar generiert werden. 

Der Transfer und die anschließende Offline-Transkription eines Audio-Streams durch die Vosk-Engine erfordern naturgemäß mehr Verarbeitungszeit. Dennoch erwies sich die Latenz bei der interaktiven Sprachsteuerung als ausreichend gering, um einen natürlichen Kommunikationsfluss mit dem System aufrechtzuerhalten, ohne dass es zu störenden Verzögerungen für den Nutzer kommt.

\subsection{Auswertung der kinetischen Interaktion (Tap-Detection)}
Die Zuverlässigkeit der Double-Tap-Erkennung über die IMU (LSM6DSOX) wurde in einer Reihe qualitativer Praxistests evaluiert. Um das System auf Robustheit zu prüfen, wurden bewusste Doppeltipp-Eingaben mit typischen Störbewegungen (wie schnelles Gehen, plötzliche Kopfdrehungen und Erschütterungen) abgeglichen.

Die Auswertung der Testreihen ergab, dass der im Code definierte Schwellenwert der Differenzbeschleunigung ($\Delta g > 1,75$) eine effektive Balance bietet. Normale Kopfbewegungen und Erschütterungen werden adäquat herausgefiltert, wodurch Fehlalarme (False-Positives) im praktischen Einsatz ausblieben. Gleichzeitig registriert das System beabsichtigte Eingaben zuverlässig, ohne dem Anwender eine unverhältnismäßig hohe Schlagkraft auf das Brillengestell abzuverlangen.

\subsection{Energetische Evaluierung (Akkulaufzeit)}
Da standardisierte Messungen unter exakt reproduzierbaren Laborbedingungen im Rahmen dieses Prototyps nicht realisierbar waren, stützt sich die energetische Evaluierung auf das beobachtete Systemverhalten im Testbetrieb.

Die theoretische Maximallaufzeit des reinen Mikrocontrollers (ausgehend von den Datenblattwerten des Herstellers) dient lediglich als grober Richtwert. In der realen Anwendung wird das System durch die kontinuierliche Funkübertragung im lokalen WiFi-Netzwerk (TCP-Stream) sowie die simultane Auslesung zusätzlicher Sensorik (Time-of-Flight-Laser und IMU) deutlich stärker belastet.

Die qualitativen Feldtests bestätigen jedoch, dass die gewählte parallele Akkukonfiguration das System stabil mit der benötigten Betriebsspannung versorgt. Spannungseinbrüche, die zu einem plötzlichen Neustart des Mikrocontrollers führen könnten, wurden während der regulären Testläufe nicht beobachtet. Das Energiemanagement erweist sich somit als ausreichend dimensioniert, um die Funktionalität des Prototyps im Rahmen der Demonstrationsszenarien verlässlich aufrechtzuerhalten.

\subsection{Einfluss der Klassenanzahl auf die Modell-Konfidenz}
Die Evaluierung der Bilderkennung analysiert die Leistungsfähigkeit der trainierten neuronalen Netze anhand der von Edge Impulse generierten Validierungsdaten. Ein direkter Vergleich der Wahrheitsmatrizen (Confusion Matrix) und der F1-Scores zweier unterschiedlicher Modell-Iterationen verdeutlicht den massiven Einfluss der Klassenanzahl auf die Erkennungsgenauigkeit. Die Gegenüberstellung der grafischen Trainingsergebnisse (welche direkt aus der Edge Impulse Entwicklungsumgebung exportiert wurden) zeigt eine eindeutige Tendenz:

\begin{itemize}
    \item \textbf{Multiklassen-Modell:} Ein Modell, das auf mehrere spezifische Klassen (wie „Beer“, „Can“, „Face“ und „Person“) trainiert ist, erreicht laut den Software-Auswertungen einen F1-Gesamtscore von lediglich 69,5\,\%. Die Confusion Matrix belegt hierbei eine hohe Fehlerquote bei der Zuweisung. Beispielsweise ordnet der Algorithmus die Klasse „Person“ in 52,2\,\% der Fälle und die Klasse „Can“ in 49,4\,\% der Fälle fälschlicherweise dem Hintergrund (Background) zu.
\end{itemize}

Diese Informationsverdichtung reduziert die Trennschärfe zwischen den einzelnen Objekten drastisch und führt zu einer wesentlich höheren Verwechslungsgefahr.

\subsection{Evaluierung komplexer Standard-Architekturen (YOLO)}
Die Validierungsphase schloss zusätzlich die Evaluierung wesentlich komplexerer YOLO-Modelle ein, um die theoretische Leistungsfähigkeit fortschrittlicherer Architekturen zu prüfen. Diese leistungsstarken neuronalen Netze durchliefen die Bildtests jedoch ausschließlich auf einem Desktop-PC. Ein Transfer der fertigen YOLO-Modelle auf die Nicla Vision scheiterte erwartungsgemäß an den strengen Hardware-Restriktionen. Dieser Testaufbau bestätigte eindrücklich, dass ein produktiver Einsatz der unmodifizierten YOLO-Architektur direkt auf der Smart-Brille technisch unmöglich ist und validierte damit den gewählten, extrem reduzierten FOMO-Ansatz als bestmögliche Alternative.
